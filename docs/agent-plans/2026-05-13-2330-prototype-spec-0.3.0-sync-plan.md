## 5. Proposed approach — three tiers, owner picks scope

### Tier 1 — Mock-layer hygiene (safe, recommended)

Type-level + mock-API alignment to the new spec. Zero UI-visible change. Cheap. No product decisions.

**T1.1 — `lib/types.ts` schema updates**
- `Chat.status` add `'awaiting_approval'`.
- `ApprovalDecisionAccepted.status` widen `'queued' → 'queued' | 'recorded'`.
- `ApprovalRequest.run_id` → `string | null`; add `chat_id: string | null`.
- Keep `requested_by_name` as a MOCK-ONLY field with a flagged comment (real backend doesn't return it; UI integration will fetch via `getUser`). Don't delete it yet — UI depends on it in 6 places.
- Add `ChatStreamFrame` variant: `{ event: 'suspended'; approval_id: string; tool: string; tool_call_id: string }`.
- Replace existing `{ event: 'error'; kind: 'approval_required'; ... }` variant — `approval_required` is the OLD pre-0.2.0 way; new spec uses the dedicated `suspended` frame. Mock streamer currently emits `error/approval_required` (verify and audit).
- Add `UserList`, `AgentVersionList`, `PatchAgentRequest`, `PatchAgentStatusRequest` types.

**T1.2 — `lib/api.ts` new mock endpoints**
- Verify `listUsers` shape matches `UserList` envelope (`{ items, total, limit, offset }`); current returns bare `User[]`. Update.
- Add `getUser(userId)`.
- Add `patchAgent(agentId, request)` — mutates the fixture's name/description/owner.
- Add `patchAgentStatus(agentId, request)` — mutates the fixture's status. Should support transitions: `draft→active`, `active→paused`, `paused→active`, `active→archived`. Reject illegal transitions with a 409-like error.
- Add `listAgentVersions(agentId, filter)` — paginated wrapper around existing version data.
- Add `getAgentVersion(agentId, versionId)`.
- Verify `listRuns` matches the spec envelope and supports `?status=` filter.

**T1.3 — `lib/fixtures.ts` polymorphic approvals**
- Add at least 1 approval row with `chat_id` populated, `run_id` null — so the mock data exercises the polymorphic shape. Other 6 keep `run_id`-anchored.

**T1.4 — Mock chat streamer**
- Update `sendChatMessage` mock generator: when a tool call hits an approval gate, emit `{ event: 'suspended', approval_id, tool, tool_call_id }` instead of (or in addition to) the current `{ event: 'error', kind: 'approval_required' }`. Flip the in-memory chat to `status: 'awaiting_approval'`.

**T1.5 — Doc fixes**
- `CLAUDE.md` — fix two URL typos (`stage.api.int3grate.ai` → `api.stage.int3grate.ai`). Update the "Key gaps (validated against live spec 2026-05-01)" block:
  - Strike the `GET /users` and `PATCH /agents/{id}` gap entries.
  - Verify and remove the stale `GET /approvals/{id} — absent` claim (it was in 0.1.0 too).
  - Add the polymorphic-approval and chat-suspension semantics as new entries.
- `docs/backend-gaps.md` — same reconciliation, propagated through every § that referenced a now-closed gap.

**Verification:** `npm run lint && npm run build` clean. No UI screens render any differently. Type errors flushed.

**Scope:** ~6 files touched, ~150 lines net. One work cycle.

### Tier 2 — UI surface restoration (decision required per surface)

Each of these was hidden specifically because an endpoint was missing. The endpoint now exists. Restoring is **a product decision, not a technical follow-on** — the surface was also screened for MVP fit at the time it was hidden.

**T2.1 — Approver / owner / version-author names**
- *Closed gap:* `GET /users/{id}` now exists.
- *Restore:* fetch user names in `ApprovalDetailScreen`, `AgentDetailScreen` (Settings + Overview), `RunDetailScreen` "actor" fields.
- *Architecture decision:* one-shot `GET /users` at app boot + keep a `Map<userId, User>` in context, OR per-row `GET /users/{id}` calls. Strong recommendation: the map approach — list pages with 50 approvals would otherwise fire 50 user lookups.
- *Risk:* low. Adds a small ContextProvider; screens stay structurally identical.
- *Push back:* the prior memory note "ALL user-name surfaces removed" suggests this was intentional UX simplification, not just a backend hack. **Worth re-confirming with the owner why names were stripped** before re-adding them. If it was just "no endpoint" — fine, restore. If it was "we don't want to over-attribute machine actions to people" — leave it.

**T2.2 — Agent rename / re-owner**
- *Closed gap:* `PATCH /agents/{id}` now exists for `name / description / domain_id / owner_user_id`.
- *Restore:* a "Settings → Identity" or just an inline rename on Agent overview.
- *Push back:* Settings page itself is currently hidden — restoring the surface raises a bigger question about whether Settings is in MVP scope at all. If we want rename without Settings, an inline edit on AgentDetail header is the smaller surface area.

**T2.3 — Pause / Resume / Archive (was "Manage employment")**
- *Closed gap:* `PATCH /agents/{id}/status` now exists with the exact 4-state lifecycle the UI was designed against.
- *Restore:* the deleted "Manage employment" card on Settings, OR a smaller "Pause employment" action on the agent overview.
- *Push back:* this surface has the same Settings-page question as T2.2. Also, "Archive" was renamed to "Fire" in user-facing UX language per `docs/ux-spec.md § 8` — confirm we keep that copy in the restoration.

**T2.4 — Version list (history)**
- *Closed gap:* `GET /agents/{id}/versions` paginated list now exists; single version too.
- *Restore:* an AgentDetail → Versions tab (currently AgentDetail has Overview / Talk / Grants / Activity / Settings / Advanced — Versions could slot in).
- *Push back:* the existing "create new version" flow lives at `/agents/:id/versions/new`. A read-only list isn't very valuable unless the user actually wants to roll back to an old version. Worth asking what end users would DO with version history.

**T2.5 — `/runs` paginated listing**
- *Closed gap:* `GET /runs` is now a real endpoint.
- *Verify, don't restore:* `RunsScreen` (the `/activity` route) already exists and lists runs. The mock probably synthesises them from chats+approvals; once we point at real `/runs`, the shape should match (`RunsList`). Lower-confidence — needs a focused check.

**Scope per surface:** 1–3 files each. Owner picks which to restore.

**Verification:** lint + build clean. Visual check on each restored route. Confirm no banned vocab leaks back in (Settings page restoration risks "Configure" / "Workflow" copy if any old strings come back).

### Tier 3 — Chat-side approvals (separate plan needed)

ADR-0011 (spec 0.2.0) introduced polymorphic approvals — `ApprovalRequest` is anchored on either a run or a chat. When an approval-gated tool fires mid-chat:
- Chat flips to `status: 'awaiting_approval'`.
- The SSE stream emits a final `event: 'suspended'` frame and ends.
- The latest `chat_messages` row carries the `approval_id`.
- Client must poll `GET /approvals/{id}` for resolution, then `GET /chat/{id}/messages?after=<last>` for the resumed turn.
- `POST /chat/{id}/message` returns 409 while chat is suspended.

**This is a real product feature, not a mock shape update**. To support it, we'd need:
- New chat-panel state: "Waiting for your decision" inline card inside the chat with Approve/Reject buttons (or a link out to the Approval).
- Approval-card variant for `chat_id`-anchored approvals — context shows "from chat with <Agent>" instead of "from run #XXXX".
- Approval list: filtering / iconography for chat-source vs run-source approvals.
- Resume mechanism: after Approve/Reject from inside chat, the chat should re-stream the resumed turn (mock streamer needs a resumption code path).
- `awaiting_approval` empty-state for the `POST /chat/{id}/message` 409 — UI shouldn't let the user type while suspended.

**This is the most impactful change in the spec bump and the one with the most product surface area.** It deserves its own plan, design walkthrough, and UX decisions about where chat-side approvals fit alongside run-side ones.

**Recommendation:** do Tier 1 first (so the mock CAN model chat-side approvals even if UI doesn't surface them yet), then have a focused conversation about Tier 3 scope. Don't bundle Tier 3 into this sync — it would balloon the change and bury the real product decisions.

## 6. Risks and trade-offs

- **Restoration drift.** Every hidden surface was hidden for a specific reason at the time. Restoring "because the endpoint exists" risks bringing back screens the owner already decided to cut for MVP. Mitigation: explicit per-surface confirmation in Tier 2.
- **`requested_by_name` lock-in.** Today's UI depends on this denormalised field that the backend will never provide. Keeping it in the mock perpetuates a divergence that's painful at integration time. Two options: (a) leave it, document, fix at swap time — cheap now, expensive later; (b) remove it now and pre-emptively route UI through a user-lookup context — more work now, smoother integration. Owner picks.
- **Mock streamer `approval_required` semantics.** The mock currently emits `{ event: 'error', kind: 'approval_required' }` — that's the 0.1.0 way the orchestrator probably never actually used. Swapping it to `{ event: 'suspended' }` is correct against the new spec but may break any existing UI that listens for the `error/approval_required` shape. Audit needed.
- **Tier 3 sprawl.** Easy to talk yourself into "while we're in there, let's wire chat-side approvals end-to-end." Don't. Tier 3 needs its own plan with screen-level mockups.
- **CLAUDE.md / backend-gaps.md drift.** These docs were authored against 0.1.0 and now have stale claims. Tier 1.5 is the doc reconciliation; if skipped, future plans will keep referencing closed gaps as if they were still open.
